## 1. 文档定位

本文件作为当前项目后续讨论、开发和验收的主文档。

使用规则：

- 已确认内容，直接作为执行方案。
- 标记为 `【待确认】` 的内容，不直接进入开发承诺，需逐条讨论后再落地。
- 原始来源仍保留 `PRD.md` 与 `1.md`，但后续不再并行作为执行口径。

### 1.1 开发流程角色

- 产品负责人：以本文件为唯一口径，确认范围、规则和验收标准。
- 实现负责人：直接修改 `excel.html`，完成功能开发、边界处理和文案收口。
- 测试负责人：负责样例文件测试、浏览器功能验证、异常路径验证。

1. 先按本文件确认开发范围与已定规则。
2. 先做规则性差异修正，再做能力增强项。

4. 全部开发完成后，按 5 个模式逐项验收。
5. 验收结论和剩余风险统一回填到本文件。

## 2. 已确认的统一方案

### 2.1 产品定位

- 产品形态：纯前端、本地浏览器、单页面 HTML 工具。
- 运行方式：无需部署，用户直接打开 HTML 文件即可使用。
- 数据安全：文件仅在本地处理，不上传服务器。
- 目标用户：以普通办公用户为主，优先服务 Excel 拆分与合并场景。

### 2.2 本期范围

本期固定保留 5 个核心模式：

1. 按工作表拆分
2. 按列拆分（横向）
3. 按列拆分（竖向）
4. 多文件合并
5. 工作表数据合并

本期不做：

- PDF
- 云端能力
- 账号体系
- 协作能力
- 多语言
- 快捷键
- 错误日志导出

后续评估项：

- 结果预览
- 样例文件
- 真实进度
- 可取消处理
- 大文件预警
- 操作帮助

### 2.3 统一交互流程

界面固定保留 4 步：

1. 上传文件
2. 选择内容
3. 配置选项
4. 生成结果并下载

界面固定保留以下区域：

- 标题区
- 模式切换区
- 步骤进度区
- 上传区
- 选择区
- 配置区
- 结果区
- 提示消息区

结果区必须展示：

- 生成文件数量
- 总大小
- 文件列表
- 单个下载
- 全部下载
- 重新开始

### 2.4 输入输出规则

- 输入支持：`.xls`、`.xlsx`
- 输出支持：`.xlsx`、`.zip`
- 不支持：`.csv`、`.xlsm`、其他格式
- `.xls` 输入统一先转换为 `.xlsx` 处理链路，再进入后续拆分或合并流程。

### 2.5 验收分层

#### P0 - 硬要求

- 数据内容正确
- 表头保留正确
- 工作表顺序正确
- 命名规则正确
- 分组逻辑正确
- 合并逻辑正确

#### P1 - 尽量保留

- 单元格样式
- 列宽
- 行高
- 日期格式
- 数字格式

#### P2 - 不作为验收要求

- 图片
- 图表
- 条件格式
- 数据验证
- 宏
- 工作表保护
- 外部连接

### 2.6 复杂 Excel 特性口径

- 公式：不计算公式；导出时以已有显示值、缓存值或原始单元格内容为主，不承诺公式行为与原文件一致。
- `.xls`：继续支持导入处理，但统一先转换为 `.xlsx` 再处理；转换后仍不承诺与原始 `.xlsx` 输入同级别保真。
- 合并单元格：不作为本期承诺能力。
- 批注：尽量保留，但不作为强验收项。
- 条件格式 / 数据验证：不作为本期承诺能力。

### 2.7 命名规则

- 不开放用户自定义命名模板。
- 命名信息仅允许使用：原文件名、工作表名、列名、分组值、日期。
- 冲突统一自动追加序号。
- 序号从 `01` 开始。
- 非法字符替换为下划线。
- 工作表名遵守 Excel 31 字符限制。
- 中文命名保留。

### 2.8 提示规则

- 所有提示统一使用中文。
- 错误后尽量保留当前用户选择。
- 至少覆盖：格式错误、读取失败、解析失败、数据为空、未完成必要选择、合并无结果、下载失败。

### 2.9 浏览器兼容口径

- 支持 Chrome / Edge 现代可用版本。
- 最低浏览器版本以测试结果为准。
- 不对 Windows 7 下旧版浏览器做强承诺。

## 3. 5 个模式的统一定义

### 3.1 按工作表拆分

- 单文件输入。
- 支持多选工作表，默认全选。
- 空工作表跳过并提示。
- 输出 ZIP。
- 目标命名：`原文件名_工作表名.xlsx`

### 3.2 按列拆分（横向）

- 单文件输入。
- 单次仅允许选择 1 个工作表。
- 必选：表头行数、拆分列。
- 空值参与分组，统一分组名为 `空`。
- 输出 ZIP。
- 目标命名：`原文件名_分组值.xlsx`

### 3.3 按列拆分（竖向）

- 单文件输入。
- 单次仅允许选择 1 个工作表。
- 必选：表头行数、固定列、数据列。
- 固定列支持多选，不允许收缩为单固定列方案。
- 每个结果文件 = 多个固定列 + 1 个数据列。
- 输出 ZIP。
- 目标命名：`原文件名_数据列名.xlsx`

### 3.4 多文件合并

- 支持多文件。
- 支持多工作表选择。
- 每个来源工作表作为独立 sheet 进入结果文件。
- sheet 重名时自动处理。
- 输出单个 `.xlsx`。
- 目标命名：`首文件名称_日期时间.xlsx`

### 3.5 工作表数据合并

- 支持多文件、多工作表。
- 以第一张工作表作为基准表头。
- 只保留一次表头。
- 统一强制添加来源列，来源值使用 `文件名-工作表名`。
- 不做分隔样式和分隔文本配置。
- 输出单个 `.xlsx`。
- 目标命名：`首文件名称_日期时间.xlsx`

## 4. 当前实现已确认差异

以下内容已确认与统一方案不一致，后续需要处理：

- 横向拆分空值当前仍使用 `未分类`，应改为 `空`。
- 按工作表拆分当前输出命名仍是 `原文件名-拆-工作表名.xlsx`。
- 多文件合并当前输出名仍是 `merged_files.xlsx`。
- 工作表数据合并当前输出名仍是 `merged_sheet_data.xlsx`。
- 工作表数据合并当前仍保留分隔样式和分隔文本逻辑。
- 工作表数据合并当前来源列仅写文件名，未写 `文件名-工作表名`。
- 页面文案里存在“所有格式将完整保留”这类过强承诺。
- 当前未形成统一的文件大小、文件数量、工作表数量、处理时长边界校验机制。
- 当前竖向拆分只支持 1 个固定列，需改为支持多固定列。
- 当前 `.xls` 虽已存在转换处理，但仍需按统一方案明确为标准处理路径，并校对转换后的行为说明。
- 当前未引入 `ExcelJS`，但后续增强方案需要同步评估其接入方式与收益。

## 5. 已确认决策

以下事项已经确认，后续按此执行：

### 5.1 性能边界策略

- 单文件 20MB
- 总量 100MB
- 最多 50 文件
- 最多 200 工作表
- 按建议方案执行：超限后优先给出明确风险提示，不直接收缩产品能力定义。
- 具体实现时，需要补充是否在个别高风险场景做前置拦截。

### 5.2 竖向拆分固定列能力

- 不能收缩为单固定列方案。
- 产品定义保持“固定列可多选”。
- 当前代码仅支持 1 个固定列，因此这里是明确的功能补齐项。

### 5.3 工作表数据合并来源列策略

- 统一强制添加来源列。
- 不再保留“是否添加来源列”的开关。
- 来源列内容统一为 `文件名-工作表名`。

### 5.4 `.xls` 处理策略

- `.xls` 继续作为输入支持格式。
- 进入处理流程前，统一先转换为 `.xlsx`。
- 方案文档明确：`.xls` 支持输入，但格式保留能力弱于原生 `.xlsx` 输入。

### 5.5 `ExcelJS` 评估策略

- 方案目标优先于技术方案。
- 为了完成能力目标，`ExcelJS` 不再视为“以后再说”的独立话题，而是同步纳入方案评估。
- 当前阶段既要推进规则对齐，也要同步评估 `ExcelJS` 引入的必要性、收益和接入成本。

当前评估结论：

- 本轮先不引入 `ExcelJS`，继续以 `SheetJS + JSZip` 完成已批准方案。
- 当前样例验证已覆盖 P0 主流程，暂未出现必须立即引入 `ExcelJS` 才能完成本轮需求的证据。
- 若后续真实业务文件在 `.xlsx` 样式、批注、数据验证保留上出现明显不足，再单独立项评估 `ExcelJS` 接入。


## 6. 建议处理顺序

建议按以下顺序逐项处理：

1. 先处理 `5.1` 性能边界策略
2. 再处理 `5.2` 竖向拆分固定列能力
3. 再处理 `5.3` 来源列策略
4. 再处理 `5.4` `.xls` 支持口径
5. 同步推进 `5.5` `ExcelJS` 评估，而不是延后到所有规则对齐之后
